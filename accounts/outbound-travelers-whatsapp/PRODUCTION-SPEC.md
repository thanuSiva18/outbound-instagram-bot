## SCOPE (decided by owner)
- **Funnel = replicate the live `.in` Instagram bot exactly:** `destination → travel_date → pax →` interactive
  **Yes/No quick-assistance** button. Persona **Rahul**, simple-English replies. **No budget, no email, no trip-type.**
- **Phone is auto-parsed** from the WhatsApp sender — never asked.
- **Job/career enquiries → email handoff (NOT a CRM/HRMS module):** detect career intent, collect name + role +
  brief details + resume, then **email `hr@outboundtravellers.com` with the CV attached.**
- **Voice notes → transcribe** (OpenAI) and feed the text into the same funnel.
- **WF2 follow-up engine → build**, with the anti-double-DM guards (the old IG nudge workflow caused repeat DMs).

## ARCHITECTURE
**WF1 "Chat & Capture"** (live): Meta webhook → respond 200 instantly → **auth/signature gate** → extract/normalize
(+ auto-phone, + voice transcription, + status-webhook guard) → 3-layer dedup/ownership → **route: travel-lead vs
job-applicant** → AI agent reply (deterministic state machine overrides) → write Google Sheet (mirror) → send WhatsApp
reply (Meta Graph API). Travel leads on "Yes" → Push to CRM. Job applicants → email HR with resume.

**WF2 "Follow-up scheduler"** (build): cron every 5 min → nudge silent/under-qualified leads → hard-stop after 2
follow-ups → park opted-out/done leads far in the future → respect the 24h WhatsApp session window.

**Storage:** Google Sheet mirror (canonical for state) + CRM (Workpex/Outbound) for travel leads + HR email for applicants.

## KNOWN FAILURE MODES — hard acceptance criteria (adapted)

### A. Inbound & Security
- **A1.** Verify Meta's `X-Hub-Signature-256` (HMAC-SHA256 of the raw body with the app secret) from message #1;
  reject forged/unknown with 401 + log. Fail-open only if the secret is intentionally unset.
- **A2.** After any webhook/verify-token/signature change, **test with a real inbound** (not curl). Keep a one-flag
  kill-switch to disable the guard during an incident.
- **A3.** Update every test/CLI client to send the signature when auth is enabled.
- **A4.** Never drop by age — dedup by `message_id` so queued/old messages after downtime still process once.
- **A5.** Drop non-actionable webhooks (status/delivery callbacks, reactions, system, ephemeral). Validate sender phone
  is 8–15 digits. *(Status-guard is live: `if(!waMessage.id) return [];`.)*

### B. Dedup / Idempotency / Concurrency
- **B1.** Three layers: (1) static `seenMsg` by `message_id`; (2) **record-level idempotency by `message_id`** in the
  sheet (a `last_msg_id`/`Uniq Id` column); (3) ownership race-win (`Claim lock` → settle → `Read lock` → `Winner?`).
  *(We have (1) and (3); add (2).)*
- **B2.** Debounce greeting-only messages (~8s); never debounce substantive ones.
- **B3.** Atomic upsert-by-phone (`ig_user_id`); settle-wait before re-read; **fail-open** on dedup read errors.
- **B4.** Static-data dedup is single-worker only — document it; move guards to a shared store if n8n is ever scaled
  to queue mode.
- **B5.** `ig_user_id` (phone) unique in the sheet; periodic dedup so split rows don't ping-pong state.
- **B6.** A new lead's fast 2nd message ("hi" then "Bali, 4 of us") must not be eaten by the greeting guard.

### C. Media (voice / CV / images)
- **C1.** Never `this.getInputData()` in a Code node — use `$input.all()`. Wrap all binary passthrough in try/catch.
- **C2.** Binary doesn't auto-flow — set `out.binary` explicitly; set `fileName`/extension/`mimeType` for transcription/email.
- **C3.** Meta media: `GET /<media-id>` with the Bearer token to resolve the URL, then download with the token header;
  guard empty buffers; fall back to media-id GET if a direct URL fails.
- **C4.** Transcription output may be string|object|nested — unpack defensively; empty → `aiDegraded` + deterministic fallback.
- **C5.** If CV email/attachment fails, still complete the applicant and tell user + HR to email it directly.
- **C6.** Non-text: transcribe audio; anything else → graceful "please type it / send as a document" and always log.

### D. Reply Formatting
- **D1.** Strip markdown before send (`toWhatsApp()` removes `*`, `__`, `##`, collapses spaces). No markdown to WhatsApp.
- **D2.** 1–2 short lines per reply (prompt + hard-coded fallbacks). ≤2 emojis.
- **D3.** Never "is that correct? yes/no" — warm ack and move on. *(Matches the .in "don't over-thank" rule.)*
- **D4.** Strip emoji from the WhatsApp profile name; empty → "there"; guard `hasName` before personalizing.
- **D5.** Any package/itinerary detail goes in media captions, not the text body.

### E. State Machine / Qualification
- **E1.** **Returning-contact re-open gate:** a prior-complete contact who messages again (not opting out) → `complete=false`,
  `status=Qualifying`, next-follow-up `+10min`. Don't re-trip "complete" on the first new "hi". *(Port the `.in` 48h/greeting
  reset — the WhatsApp Normalize currently lacks it.)*
- **E2.** Gate COMPLETE on deterministic fields (`destination + travel_date + pax`), not the model's stage. *(Live: deterministic.)*
- **E3.** Update backstop: a restated date/pax overrides the stale stored value.
- **E4.** Fields independent; never re-ask a declined field. *(email out-of-scope.)*
- **E5.** Monotonic step — never decreases.
- **E6.** Run field extractors before notes capture (don't file "Bali, 4 pax" as a free-text note).
- **E7.** Capture multiple volunteered details at once; skip answered steps. *(Prompt already instructs this.)*
- **E8.** "Not interested" → warm back-off; check negation first.
- **E9.** **Opt-out** phrase-anchored ("stop", "unsubscribe", "stop messaging") — not any message containing "stop".
  On opt-out: warm ack, `status=Unsubscribed`, max follow-up counter, park far-future, **no CRM push.** *(Not implemented yet.)*
- **E10.** Hot-lead alert fires once on transition (guard `knownTemp != Hot`). *(Our "hot" = travel date within 60d and/or
  large group, since no budget — confirm with owner.)*

### F. Extraction / Normalization / Scoring
- **F1/F2/F4/F7 (budget) — OUT OF SCOPE** (no budget field). N/A.
- **F3 (email) — OUT OF SCOPE.** N/A.
- **F5.** Lead source sticky: freeze `whatsapp`; AI may only upgrade a "Direct"-equivalent, never override a known source.
- **F6.** Referring-agent name: word-boundary + explicit cue ("ask for X", "X referred me"), longest-match first; bot only
  reports it (into `assigned_to`/notes), CRM does authoritative routing.
- **Our real extractors to harden:** phone normalization (`cleanPhone`), destination canonicalization
  (`normalized_destination`), pax parsing, free-text `travel_date` — robust to typos and to **transcribed-voice** phrasing.

### G. Follow-up Engine (WF2) — to build
- **G1.** Verify the new WF2 id against this host; smoke-test executions right after deploy.
- **G2.** Never double-text mid-conversation: skip leads touched in the last N min (`last_update_ts`); first gap ~10 min.
- **G3.** One workflow owns the follow-up counter — no WF1/WF2 last-writer-wins races.
- **G4.** Respect the **24h session window**: a free-form send >24h after the user's last inbound fails — on send failure
  **do not** increment the counter; use an approved **template** message + retry.
- **G5.** Batch + jitter bulk sends (no returnAll burst after downtime).
- **G6.** Opted-out/done leads only un-stick when they message again.

### H. CRM Push Contract
- **H1.** Keep the push token == the live token exactly; log 401/402 with a masked prefix. **Known: the Outbound CRM
  (Vercel) leg returns 402 right now** — surface it loudly; don't let it look like success.
- **H2.** Not just `onError:continueRegularOutput` — retry 3× with backoff; on final failure write a visible "Bot Errors"
  health row; keep a reconciler that flags sheet rows qualified >30 min with no CRM record.
- **H3.** Upsert by digits-only phone (last 10–12 digits). Merge semantics: never overwrite a filled field with empty;
  union tags; never reset status or yank an agent assignment.
- **H4.** Map to the CRM's accepted field whitelist (see `docs/crm-integration-contract.md`); fold unmapped into a summary;
  confirm the first push actually populates fields.
- **H5.** Agentless leads created into an admin/overflow pool, never rejected.
- **H6.** Confirm active-lead cap + `assigned_sp_id` mapping after any import/rename.

### I. Applicant Routing → email handoff
- **I1.** Detect career intent **intent-first** (no travel intent present) with word-boundary keywords (job, vacancy,
  hiring, resume, cv, apply, opening, career) so a traveller mentioning "job" isn't lost.
- **I2.** Collect: name, role/position of interest, brief details (+ resume). Keep applicants "sticky" (~7 days) so a
  one-word answer doesn't flip routing back to travel.
- **I3.** Idempotent: don't email HR twice for the same applicant/message.

### J. Resume → attach to HR email
- **J1.** Resume arrives as a WhatsApp **document** (or the applicant is told to send one). Fetch via Meta media
  (`media-id → URL → download with Bearer`).
- **J2.** Email `hr@outboundtravellers.com` with applicant info + **resume attached**; dedup on `message_id`.
- **J3.** If the resume is missing/failed, still email the applicant info and ask them to reply with the CV (or email HR directly).
- **J4.** Verify end-to-end: WhatsApp CV → HR inbox receives the correct file + applicant details.

### K. Observability / Error Handling
- **K1.** Wrap every external call (Sheets, Meta, CRM, OpenAI, email, n8n API) in try/catch; degrade safely.
- **K2.** AI degraded (e.g. expired key) → set `aiDegraded` + log a high-severity health row; never run the silent fallback
  with no signal to ops.
- **K3.** Log Meta send failures to a `LastSendError` field + a monitor.
- **K4.** Critical path = fail-open (never block a reply); optional enhancements = fail-silent. Confirm `errorWorkflow`
  `f8JebCcUmgk137Li` exists and does something useful.

### L. Deploy / Infra / Credentials
- **L1.** Always target this n8n host explicitly; re-verify every workflow/credential id against it.
- **L2.** Prefer surgical partial-update (single-node) over full rebuilds; commit the build/export script **and** the
  exported workflow JSON together — never leave live edits untracked.
- **L3.** Move the **Meta token out of the inline Send header into an n8n credential** (rotation safety). The OpenAI key
  lives in both `.env` (tooling) and the n8n credential (the live bot) — editing `.env` doesn't change the bot.
- **L4.** Validate any new key read-only before use; mind credential PATCH schema quirks.
- **L5.** The n8n public-API token expires → 401; document re-issue; keep a credential changelog.
- **L6.** Delete stale/duplicate WA workflow drafts so nobody deploys the wrong one.

## ACCEPTANCE TESTS (offline regression harness — all pass before go-live)
- **Dedup:** duplicate `message_id` → exactly one reply; burst greetings collapse; substantive never debounced; fast
  2nd message merged.
- **Status webhooks:** Meta delivery/read callbacks → ignored (no AI, no send, no sheet write).
- **Returning contact:** prior-complete lead messages again → re-opens, WF2 re-arms (not parked).
- **Premature complete:** model says complete with only destination → not marked complete (needs date + pax).
- **Update backstop:** restated date/pax overrides the stale value.
- **Opt-out:** "stop" → Unsubscribed, no CRM push, WF2 never nudges; "stop asking" mid-flow ≠ opt-out.
- **Voice:** voice note transcribed → funnel proceeds; transcription empty → graceful fallback + lead logged.
- **Non-text:** image/location → "please type it" + lead logged.
- **Formatting:** every reply path: no markdown, ≤2 lines, ≤2 emojis; profile-name emoji stripped.
- **Auto-phone:** new sender → `whatsapp_number` auto-filled; never asked.
- **CRM:** same phone twice → update not duplicate; empty field never blanks a filled one; 401/402 surfaced visibly.
- **Applicant:** career intent → not treated as travel; collects info; CV emailed to HR with attachment; idempotent;
  missing CV still emails info + asks for it.
- **WF2:** silent lead nudged after ~10 min; never mid-conversation; stops after 2; >24h window → template or no-count.
- **Hot alert:** fires once on transition, routed to the matched agent (not a hardcoded number).
